# Внешний оркестратор + файлы как истина, одноразовые pi-воркеры

**Контекст.** Нужен движок «глубокого исследования» с жёстким требованием:
останавливаться в любой момент и продолжать с того же места. pi — интерактивный
TUI-агент; его механизмы (`pi --continue`, in-session fan-out через
`@tintinweb/pi-subagents`) хранят состояние внутри живой сессии.

**Решение.** State-машиной владеет отдельный долгоживущий процесс-**оркестратор**
(Python), а единственная истина — файлы в папке Run'а (`spec.md`,
`backlog.jsonl`, `state.json`, `findings/*.md`). **Worker**'ы — это одноразовые
вызовы `pi -p`: один запуск = одно задание = один Finding на диске.
Преемственность между заданиями (если нужна) передаётся не живой сессией, а
подкладыванием прежних Finding'ов в промпт (`@file`).

**Почему так, а не иначе.**
- Пауза/стоп/продолжение становятся тривиальными: убить оркестратор и Worker'ов
  безопасно, в полёте максимум один Finding, который либо лёг на диск, либо нет.
- Отвергнут **pi-как-оркестратор + pi-subagents**: resume живой сессии с N
  подагентами в полёте хрупок (оборванные fetch'и, потерянная очередь), и
  контроль из веб-морды ограничен.
- Отвергнут **пул `pi --mode rpc`**: его главный плюс (не перегружать модель) тут
  не работает — модель живёт в inference-сервере (LM Studio / Ollama / vLLM), а
  не в pi; зато пул накапливает контекст в живой сессии (теряется при kill) и
  требует reconnect/health-check, т.е. возвращает ровно ту хрупкость, которую
  убирала схема «истина на диске».
