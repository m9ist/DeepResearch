# Иммутабельные артефакты и историчность Run'а

**Контекст.** Появилось требование (Олег): уметь на паузе «докидывать вопросы/идеи»
и синтезировать из накопленных Finding'ов + уточнений НОВЫЙ план, с которым Run
продолжается ([Replan](../../CONTEXT.md)). Чтобы это было прослеживаемо, нужна
полная историчность: видно, какой план был, что человек добавил, как он
эволюционировал. Сейчас же `state.json` / `backlog.jsonl` / `spec.md` мутируются
«на месте» — старые версии теряются.

**Решение (направление, механизм ещё проектируется).** Делать артефакты Run'а
**иммутабельными/версионируемыми**: ничего не перезаписывать поверх, каждое
значимое изменение — новая версия или запись в append-only лог.
- `spec.md` → версии `spec.vN.md` + указатель на текущую.
- `backlog` → append-only лог операций (add/claim/done/replan) + воспроизводимые
  снапшоты; вместо правки строки — новая запись.
- `state` → лог переходов, а не одно перезаписываемое значение.
- `events.jsonl` уже append-only — это зачаток подхода.
- **Replan** создаёт новую версию плана поверх старой, не стирая её.

**Почему так.** Прослеживаемость (откуда взялся план, что добавил человек),
безопасный откат, и честная «остановка в любой момент» без потери прошлого.
Цена — больше файлов/логики и нужен указатель «актуальная версия»; читателям
(вьюер, синтез) надо смотреть на current, но иметь доступ к истории.

**Статус: proposed.** Текущая реализация v1 мутирует артефакты на месте
(сознательный shortcut). Переход на иммутабельность — отдельный крупный рефактор
(см. BACKLOG, phase 2), делаем когда дойдём до Replan/Pause.
