# Лестница провайдеров транскриптов с fail-over без блокировки

**Контекст.** Скачка YouTube-транскриптов ([youtube_one.py](../../tools/youtube_one.py))
держалась на одном бесплатном `youtube_transcript_api`, который YouTube банит по IP
за частые запросы. Workers — параллельные процессы, координировались общим
temp-marker троттлом (`DR_YT_MIN_INTERVAL`). Появилось требование (Олег): кэш,
несколько провайдеров с приоритетом, аккуратный темп запросов, бэкофф при бане и
лог для поиска паттернов. Ключевое напряжение: интервал «не чаще раза в 5 минут»
у бесплатного API против десятков YouTube-ссылок в одном Run'е и параллельных
Worker'ов.

**Решение.** Ввести **Provider Cascade** — упорядоченную по приоритету лестницу
**Transcript Provider**'ов ([CONTEXT.md](../../CONTEXT.md)), запрос идёт сверху вниз.

- **Fail-over без блокировки.** Провайдер, чьё ограничение сейчас исчерпано
  (кулдаун / месячная квота / бан), молча пропускается — переходим к следующему.
  Ни один Worker не ждёт на таймере. Бесплатный `youtube_transcript_api` —
  «оппортунистический» путь (дёргаем не чаще его интервала, ~5 мин), рабочая
  лошадка при его кулдауне — платный apify-актор.
- **Мелкозернистый лок.** Лок берётся только на короткую критсекцию
  «решить + записать состояние» (`last_request_ts`, `used_this_month`, `ban_level`
  в `tools/tmp/yt_state.json`) и отпускается **до** сетевого вызова. Сериализуется
  не «загрузка транскрипта вообще», а только бан-опасный путь YT-API (через захват
  5-минутного слота). apify бежит параллельно, трогая лок лишь чтобы списать квоту.
- **Бэкофф по бану.** `effective_interval = min(interval·factor^ban_level, cap)`;
  `IpBlocked/RequestBlocked/PoTokenRequired` → `ban_level+=1`, первый успех → 0.
  Прочие исключения (нет субтитров, видео недоступно) — промах без роста бэкоффа.
- **Лог вместо негативного кэша.** Все взаимодействия пишем в
  `tools/tmp/yt_runs.jsonl` (строка/попытка + строка-итог). Файлов-«фейлов» не
  создаём: видео без субтитров будет переспрашиваться, зато нет «залипших»
  негативных меток, а паттерны видны в логе.
- **Конфиг расщеплён.** `tools/yt_providers.json` — публичный, коммитим
  (лестница, `enabled`, интервалы, квоты, бэкофф, actor). Секреты —
  `tools/yt_secrets.json` за gitignore (+ `.example`). Изменяемое состояние —
  отдельный `tools/tmp/yt_state.json`, его пишет только код под локом.
- **Кэш — отдельный слой.** `tools/tmp/.yt_cache/yt-<id>/` по ключу `videoId`,
  общий для всех Run'ов; на хите копируется в `sources/` Run'а (Run остаётся
  самодостаточным).

**Почему так.** Блокирующий вариант (Worker ждёт 5-минутный слот) сериализует
транскрипт-тяжёлые Run'ы в часы и держит слоты concurrency впустую. Fail-over
сохраняет параллелизм и делает бесплатный API «бесплатной добавкой», а не узким
горлом. Мелкозернистый лок не даёт квотируемым провайдерам простаивать.
Цена: больше движущихся частей (3 файла + кэш + лог), возможный повторный перебор
лестницы на видео без субтитров, и контрактная правка — таймаут `execFile` в
[index.ts](../../extension/index.ts) поднят 120→300 c под холодный старт apify.

**Статус: proposed.** Заменяет старый temp-marker троттл (`DR_YT_MIN_INTERVAL`).
Провайдеры v1: `youtube_transcript_api` → `apify_supreme_coder`. Новых
pip-зависимостей нет (apify через stdlib `urllib`).
